
=====================================================================
1. VISÃO GERAL DO PROJETO
=====================================================================

O projeto SAEP Analytics é uma aplicação web front-end voltada à leitura, processamento e visualização de dados pedagógicos provenientes de planilhas Excel no contexto do SAEP. A aplicação foi estruturada para importar uma planilha contendo informações de desempenho de turma e desempenho individual por registro, transformar esses dados em estruturas internas padronizadas e exibir:

1. Indicadores gerais da turma
2. Desempenho por capacidade
3. Matriz/ranking dos alunos
4. Boletim individual do aluno
5. Gráficos por capacidade e por conhecimento
6. Diagnóstico pedagógico textual
7. Exportação em PDF do aluno
8. Exportação em PDF da turma
9. Impressão do boletim individual

A reorganização técnica realizada neste projeto teve como foco principal separar responsabilidades, mantendo comportamento, estilo visual e praticidade de uso. A intenção não foi reinventar a aplicação, mas melhorar sua estrutura para facilitar manutenção futura, novas implementações e leitura do código por terceiros.

=====================================================================
2. OBJETIVO DA REORGANIZAÇÃO
=====================================================================

Antes da reorganização, a lógica principal da aplicação estava concentrada em um único arquivo JavaScript. Esse modelo funcional resolve o problema inicial, porém apresenta limitações clássicas quando o projeto começa a crescer:

- mistura de responsabilidades em um único arquivo
- maior dificuldade para localizar bugs
- dificuldade para alterar uma parte sem impactar outra
- baixa legibilidade para novos colaboradores
- aumento do acoplamento entre DOM, regras de negócio e dados
- risco de regressão em manutenção
- dificuldade de reutilização de código
- maior custo cognitivo para compreender fluxos

A reorganização teve os seguintes objetivos:

1. Separar responsabilidades por contexto funcional
2. Isolar a regra de negócio da camada de interface
3. Reduzir acoplamento entre manipulação do DOM e cálculo de métricas
4. Tornar o sistema mais inteligível para manutenção assistida por IA
5. Preservar as integrações existentes com HTML, CSS e bibliotecas externas
6. Manter a experiência do usuário sem alteração perceptível
7. Preparar a base para futuras refatorações mais profundas

=====================================================================
3. METODOLOGIA UTILIZADA NA REORGANIZAÇÃO
=====================================================================

A metodologia aplicada na reorganização seguiu princípios de Clean Code, separação de responsabilidades e refatoração incremental sem ruptura funcional.

3.1. Estratégia geral

A abordagem escolhida foi:
- manter o sistema 100% em JavaScript tradicional carregado por script tags
- evitar introdução prematura de bundlers ou módulos ES complexos
- separar o código em arquivos menores com coesão temática
- respeitar as dependências entre arquivos por ordem de carregamento
- preservar os IDs e estruturas principais do HTML
- manter o CSS funcional sem necessidade imediata de revisão
- garantir compatibilidade com as bibliotecas já utilizadas

3.2. Critérios de separação dos arquivos

A divisão dos arquivos foi feita com base em responsabilidade primária:

- estado da aplicação
- constantes e configuração derivada
- utilidades puras
- persistência em localStorage
- leitura/parsing da planilha
- regra de negócio SAEP
- renderização de gráficos
- renderização da visão geral
- renderização da tela individual do aluno
- geração de PDF
- controle de upload e ações do usuário
- bootstrap/inicialização do sistema

3.3. Princípios aplicados

Foram aplicados os seguintes princípios de projeto:

A. SRP — Single Responsibility Principle
Cada arquivo e cada função foram organizados para assumir um papel predominante e mais delimitado.

B. DRY — Don’t Repeat Yourself
Funções repetitivas de parsing, normalização, set de texto no DOM e recuperação de dados foram centralizadas.

C. KISS — Keep It Simple, Stupid
Foi evitado excesso de abstração desnecessária. A arquitetura permanece simples o bastante para uso imediato.

D. Nomenclatura clara
Os nomes das funções e estruturas foram escolhidos para revelar intenção e não apenas implementação.

E. Separação entre domínio e apresentação
A regra de negócio do SAEP foi deslocada para um serviço próprio, reduzindo dependência direta da interface.

F. Refatoração sem quebra de contrato
Os elementos HTML esperados pela aplicação foram preservados, assim como a semântica geral do fluxo.

=====================================================================
4. CONTEXTO FUNCIONAL DA APLICAÇÃO
=====================================================================

A aplicação possui duas páginas principais:

1. index.html
   Exibe a visão geral da turma.

2. aluno.html
   Exibe o boletim individual do aluno.

Além disso, a aplicação utiliza um arquivo config.js que define:
- cursos
- nomes de capacidades
- diagnósticos pedagógicos
- cores

A planilha importada deve conter abas específicas, com tolerância a pequenas variações de nome, permitindo que o sistema reconheça automaticamente as planilhas válidas.

7. DESCRIÇÃO DETALHADA DOS ARQUIVOS DA PASTA /js

---------------------------------------------------------------------
7.1. js/state.js
---------------------------------------------------------------------

Objetivo:
Centralizar o estado global da aplicação em um único objeto.

Conteúdo principal:
- ApplicationState

Estrutura:
- studentSummaries
- answerRecords
- selectedStudent
- charts.capacityChart
- charts.knowledgeChart

Responsabilidade:
Este arquivo existe para evitar variáveis globais soltas e manter um ponto único para dados temporários da aplicação.

Motivação técnica:
Sem esse arquivo, seria comum espalhar variáveis como lista de alunos, registros, aluno atual e instâncias de gráfico em vários pontos. Isso prejudica legibilidade, manutenção e previsibilidade.

Benefícios:
- simplifica a leitura do estado da aplicação
- facilita restauração e atualização de dados
- centraliza dependências de outras camadas

Riscos se mal utilizado:
- virar um “estado-deus” com muitos dados não relacionados
- acoplar demais o sistema a estrutura global mutável

Recomendação futura:
Se a aplicação crescer, pode-se evoluir para um padrão mais formal de gerenciamento de estado, mas no momento o objeto centralizado é suficiente e proporcional.

7.2. js/constants.js
---------------------------------------------------------------------

Objetivo:
Armazenar constantes fixas, aliases e fallback de configuração.

Conteúdos principais:
- STORAGE_KEYS
- DEFAULT_COLORS
- COLOR_CONFIG
- COURSE_CONFIG
- WorkbookSheetAliases
- SummaryColumnAliases
- RecordColumnAliases

Responsabilidade:
Remover “strings mágicas” do código e concentrar convenções de leitura e persistência em um único local.

Detalhes importantes:

A. STORAGE_KEYS
Guarda as chaves utilizadas no localStorage:
- students
- records
- course

B. DEFAULT_COLORS
Define o conjunto padrão de cores caso config.js não forneça dados.

C. COLOR_CONFIG
Seleciona as cores reais a partir de SAEP_CONFIG ou fallback.

D. COURSE_CONFIG
Seleciona a configuração de cursos carregada do config.js.

E. WorkbookSheetAliases
Lista nomes possíveis para as abas esperadas da planilha.

F. SummaryColumnAliases
Lista nomes possíveis das colunas da aba resumo.

G. RecordColumnAliases
Lista nomes possíveis das colunas da aba por registro.

Valor arquitetural:
Esse arquivo reduz o impacto de mudanças de nomenclatura e torna o parsing mais robusto.

Benefícios:
- manutenção concentrada
- maior tolerância a variações de planilha
- menor repetição

7.3. js/utils.js
---------------------------------------------------------------------

Objetivo:
Concentrar utilidades puras e helpers genéricos não específicos do domínio SAEP.

Funções presentes:
- getElementByIdOrNull
- setTextContentIfElementExists
- setHtmlContentIfElementExists
- normalizeText
- normalizeHeaderName
- parseNumber
- parseCapacityCode
- sanitizeFileName
- convertTimeToMinutes
- calculateAverageTime
- setButtonLoadingState
- restoreButtonState

Responsabilidade:
Fornecer pequenas funções reutilizáveis para:
- acesso ao DOM
- normalização
- parsing
- formatação operacional
- controle visual de loading em botões

Descrição detalhada das funções:

1. getElementByIdOrNull(elementId)
Busca um elemento por ID no DOM.
Retorno:
- HTMLElement quando existe
- null quando não existe

Uso:
Usado em praticamente toda a camada de UI.

2. setTextContentIfElementExists(elementId, value)
Seta innerText com segurança, apenas se o elemento existir.

3. setHtmlContentIfElementExists(elementId, htmlContent)
Seta innerHTML com segurança, apenas se o elemento existir.

4. normalizeText(value)
Converte qualquer valor para string limpa e removida de espaços laterais.

5. normalizeHeaderName(headerName)
Normaliza strings de cabeçalho:
- remove acentos
- reduz espaços
- converte para minúsculas

6. parseNumber(value)
Transforma strings numéricas da planilha em número.
Suporta:
- porcentagem
- vírgula decimal
- vazio

7. parseCapacityCode(rawValue)
Padroniza valores de capacidade para o formato C1, C2, C3 etc.

8. sanitizeFileName(fileName)
Remove caracteres problemáticos para uso em nome de arquivo exportado.

9. convertTimeToMinutes(timeValue)
Converte tempo em formatos aceitos para minutos.

10. calculateAverageTime(timeValues)
Calcula média de tempos válidos e retorna string formatada.

11. setButtonLoadingState(buttonElement, loadingHtml)
Ativa estado visual de carregamento no botão.

12. restoreButtonState(buttonElement, originalHtml)
Restaura estado original do botão.

Valor arquitetural:
Este arquivo diminui repetição e mantém as demais camadas mais enxutas.

7.5. js/spreadsheet-service.js
---------------------------------------------------------------------

Objetivo:
Ler a planilha importada, encontrar as abas corretas, interpretar colunas e retornar objetos estruturados.

Funções presentes:
- findWorksheetByPossibleNames
- getColumnValue
- buildStudentSummary
- buildAnswerRecord
- validateImportedSpreadsheetData
- readSpreadsheetData

Responsabilidade:
Converter o conteúdo bruto do arquivo Excel em dados internos confiáveis.

Descrição das funções:

1. findWorksheetByPossibleNames(workbook, possibleNames)
Procura uma aba do workbook com base em lista de nomes possíveis normalizados.

2. getColumnValue(rowData, possibleColumnNames)
Procura um valor em um objeto-linha considerando múltiplos nomes alternativos de coluna.

3. buildStudentSummary(summaryRow)
Converte uma linha da aba resumo em um objeto padronizado de aluno.

Campos gerados:
- nome
- matricula
- desempenho
- acertos
- erros
- tempo
- proficiencia
- nivelSAEP

4. buildAnswerRecord(recordRow)
Converte uma linha da aba de registros em um objeto analítico.

Campos gerados:
- aluno
- matricula
- cap
- conhecimento
- acertou

5. validateImportedSpreadsheetData(studentSummaries, answerRecords)
Valida o resultado da importação e lança erro em caso de ausência de conteúdo relevante.

6. readSpreadsheetData(fileBuffer)
Função orquestradora da leitura da planilha.
Etapas:
- cria workbook via XLSX
- encontra abas esperadas
- transforma abas em JSON
- converte cada linha em objeto interno
- valida o resultado
- retorna estrutura final

Valor arquitetural:
Este arquivo isola completamente a complexidade da planilha e protege o restante da aplicação do formato bruto do Excel.

Ponto forte:
A tolerância a aliases de nomes de abas e colunas torna o sistema mais robusto.

- inicializa página do aluno, se necessário

Valor arquitetural:
Evita que cada página precise de um script totalmente diferente. O bootstrap central decide o comportamento pela presença de elementos.

=====================================================================
8. FLUXO OPERACIONAL DA APLICAÇÃO
=====================================================================

8.1. Fluxo de carregamento
1. O navegador carrega config.js
2. Os arquivos JS são carregados em ordem
3. app.js executa initializeApplication()
4. O sistema:
   - lê dados do localStorage
   - popula o seletor de curso
   - ativa o upload
   - detecta a página atual
   - renderiza a interface correspondente

8.2. Fluxo de upload
1. Usuário seleciona planilha
2. upload-controller.js captura o evento
3. spreadsheet-service.js lê o arquivo
4. dados são transformados em estruturas internas
5. ApplicationState é atualizado
6. dados são persistidos no localStorage
7. a página é recarregada

8.3. Fluxo da visão geral
1. dashboard-ui.js solicita estruturas agregadas à camada de negócio
2. renderiza KPIs
3. renderiza cards por capacidade
4. renderiza tabela/ranking

8.4. Fluxo da página do aluno
1. student-ui.js popula o seletor
2. usuário escolhe um aluno
3. o sistema atualiza o cabeçalho
4. gera os gráficos
5. monta o diagnóstico pedagógico

8.5. Fluxo de PDF
1. usuário clica em PDF
2. o botão entra em loading
3. o pdf-service.js executa a geração
4. o arquivo é salvo
5. o botão retorna ao estado normal

11. DECISÕES ARQUITETURAIS IMPORTANTES
=====================================================================

1. Não migrar imediatamente para módulos ES
Motivo:
A prioridade foi reorganizar sem introduzir complexidade de build, bundler ou ambiente adicional.

2. Manter script tags com ordem controlada
Motivo:
Maior praticidade para uso imediato.

3. Preservar IDs do HTML
Motivo:
Evitar quebra de contrato entre front e JS.

4. Manter CSS intacto na etapa atual
Motivo:
O problema estrutural principal estava no JavaScript.

5. Centralizar regra de negócio em saep-service.js
Motivo:
Evitar espalhamento de lógica pedagógica.

6. Centralizar parsing da planilha em spreadsheet-service.js
Motivo:
Separar formato bruto do dado da regra de negócio.

Ao continuar a evolução desse projeto no Claude Code, considere as seguintes diretrizes:

1. Manter a separação de responsabilidades atual
2. Evitar recolocar parsing, UI e regra no mesmo arquivo
3. Preservar a ordem dos scripts caso continue em script tags
4. Ao alterar o HTML, verificar todos os IDs esperados

7. Ao alterar PDF, priorizar pdf-service.js
